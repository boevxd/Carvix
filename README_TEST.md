## 4. Что именно проверяется

### 4.1. Безопасность

- ✅ Пароль хешируется через **bcrypt** (sanity-check: пароль из БД никогда не попадает в HTTP-ответ).
- ✅ JWT подписан **HS256** с серверным секретом; tampering payload-а отвергается с `InvalidSignatureError`.
- ✅ Просроченный токен → `TokenExpiredError`.
- ✅ Токен с чужим секретом отвергается.
- ✅ Все защищённые эндпоинты возвращают **401** без токена.
- ✅ RBAC чётко различает три уровня доступа: read-only (Аналитик), read+write по операциям (Главный механик), полный доступ + управление бюджетами (Директор).
- ✅ `limit` параметра пагинации обрезается до **500** (защита от полной выкачки таблицы DoS-запросом).

### 4.2. Бизнес-правила финансового модуля

- ✅ Категории расходов ограничены whitelist'ом: `remont`, `zapchasti`, `topliv`, `strakhovka`, `nalog`, `moyka`, `prochee`.
- ✅ Сумма расхода и плановая сумма бюджета проверяются на положительность.
- ✅ Месяц бюджета — в диапазоне 1–12.
- ✅ Формула план/факт работает корректно во всех сценариях: 100% исполнения, недоисполнение, перерасход, нулевой план (защита от деления на 0), округление процента до 0.1 %.
- ✅ Оптимистичное обновление склада в транзакции при приёме накладной + реверс остатков при удалении.
- ✅ Audit-log пишется при каждой write-операции с правильным `sotrudnik_id` из JWT.

### 4.3. Парсер CSV-импорта расходов

- ✅ Разделитель `;` (Excel-RU) и `,` (стандартный CSV).
- ✅ Кириллические заголовки (дата/категория/сумма/гос_номер/подразделение).
- ✅ Числа с пробелами и запятой как разделителем дробной части (`"12 500,50"` → `12500.50`).
- ✅ Импорт по гос-номеру ТС или по названию подразделения.
- ✅ Подробная диагностика ошибок построчно: номер строки + причина.
- ✅ Пустой CSV (только заголовок) — 400 с корректным сообщением.

### Ключевая идея

PostgreSQL **не поднимается в тестах**. Вместо этого `db.js` заглушается через `jest.mock()` синглтон-моком (`__tests__/helpers/mockDb.js`), который:

1. Записывает все вызовы `pool.execute / pool.pool.query / transaction(tx)` в массив `__calls`.
2. По regex-паттернам SQL-запроса возвращает заранее настроенные `__when(matcher, response)` ответы.
3. После теста позволяет проверить **что и в каком порядке** было вызвано — например, что после POST-запроса действительно записан `INSERT INTO finansoviy_log` с тем же `sotrudnik_id`, что и в JWT.

Это даёт **настоящие интеграционные тесты HTTP-эндпоинтов** (со всем стеком Express + middleware + роуты + transaction wrapper) — но без сетевых задержек, без сидинга демо-данных, без зависимости от состояния таблиц. Всё **3.2 сек на 119 тестов**.

## 7. CI/CD

`.github/workflows/ci.yml` запускает все тесты на двух версиях Node одновременно (**18.x и 20.x**) при каждом push в `main` и каждом pull-request:

```yaml
jobs:
  test:
    strategy:
      matrix:
        node: ['18.x', '20.x']
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
          cache: 'npm'
      - run: npm ci
      - run: npm run test:ci
      - uses: actions/upload-artifact@v4
        with:
          name: coverage-html
          path: coverage/lcov-report/
```

После прохождения CI **Render автоматически деплоит** `main` на https://carvix.onrender.com.
